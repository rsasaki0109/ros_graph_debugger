# 映え強化プラン — Claude(上司) × Codex(部下) 開発体制

> 目的: ros_graph_debugger を「一目で伝わる・思わずシェアしたくなる」ツールに
> 育てるための実行プラン。設計・指示書作成・レビューを Claude(上司)、実装を
> Codex(部下) が担当する。本ドキュメントが両者の共有コンテキストであり、
> 各タスクの指示書はここからコピーして Codex に渡す。

最終更新: 2026-06-11

## 1. 体制と運用ルール

### 役割分担

| 役割 | 担当 | 具体的な仕事 |
|------|------|--------------|
| 上司 | Claude | アイデア立案、仕様設計、指示書作成、受け入れ基準の定義、コードレビュー、デモ素材(GIF/動画)の演出設計 |
| 部下 | Codex | 指示書に基づく実装、テスト追加、`rgd serve --demo` での動作確認、セルフレビュー後の差分提出 |

### 指示書フォーマット（必須5項目）

Codex への依頼は必ず以下の形式で渡す。曖昧な依頼は事故のもとなので、
1項目でも欠けたまま渡さない。

1. **背景** — なぜやるのか。映えポイントと既存資産との関係。
2. **タスク** — 何を作るか。対象ファイルを明記。
3. **制約** — 依存追加の可否、性能要件、互換性要件。
4. **受け入れ基準** — 「`rgd serve --demo` でこう見えること」レベルまで具体化。
5. **禁止事項** — 触ってはいけないファイル、変えてはいけないスキーマ。

### レビューフロー

1. Codex がタスクごとにブランチを切り実装（ブランチ名: `feat/<task-id>-<slug>`）。
2. 差分提出時に必ず添付するもの:
   - `python -m pytest test/` の結果（全テストパス必須）
   - `rgd serve --demo` のスクリーンショットまたは画面収録（UI変更時）
3. Claude がレビュー。観点は各指示書の「レビュー観点」に列挙。
4. NG なら指摘リストを返して再提出。2往復で収束しないタスクは仕様を疑い、
   指示書側を直す（部下を責める前に指示を疑うこと）。

### 環境上の注意（重要）

- **この開発環境ではクロスプロセスの DDS discovery が壊れている。**
  マルチプロセスの ROS デモは互いを発見できないので、動作確認は
  `rgd serve --demo`（DDS 不要のスクリプトリプレイ）か、インプロセスの
  テストで行うこと。実機・複数ノード前提の検証はタスクの受け入れ基準に
  含めない。
- コミットに Co-Authored-By は付けない。PR 説明文に AI 生成表記を入れない。

## 2. 現状資産の棚卸し（2026-06-11 時点）

映え系はすでにかなり積んである。新タスクはこれらの「上に」乗せる。

- **ライブデモ (GitHub Pages)** — 実 Web UI がスクリプト済みボトルネックデモを
  完全クライアントサイドで再生。README からワンクリック。
- **アニメ GIF ヒーロー** — 実デモを収録した demo.gif が README 冒頭に鎮座。
- **グラフ上のデータフローアニメ** — エッジに沿ってデータ粒が流れる演出済み。
- **ソーシャルカード** — SVG ヒーロー + OG 画像。
- **`rgd diff`** — 2つの録画間の回帰検出。`--fail-on-regression` で CI ゲート可。
- **federation** — `rgd federate robot1=URL ... --serve` で複数エージェントを
  1画面にマージ表示（ホスト名前空間付き）。
- **タイムラインスクラバ + 録画再生** — `rgd serve run.rgd.json`。
- **pipeline-path tracer** — `/api/v1/path?target=` で律速経路をグラフ上で点灯。
- **Web スタック** — `web/app.js`(約1000行) + cytoscape/dagre (vendor 済み)。
  ビルドステップなし、素の JS。

## 4. 指示書

### T1: エッジ上のライブ心電図（EKG スパークライン）

**背景**
既存のデータフローアニメは「流れている」ことは伝わるが「どのくらいの
レートか」「いつ詰まったか」が絵にならない。各エッジに直近のトピック
レートをミニ波形（心電図風）で描けば、ボトルネック発生の瞬間に波形が
平坦化して赤くなる——ライブデモが一段「生きてる」感じになる。
小粒タスクなので、Codex との協業フローの試運転を兼ねる。

**タスク**
`ros_graph_debugger/web/app.js` のエッジ描画に、トピックごとの直近 30 秒の
レート履歴をスパークラインとしてエッジ中点付近にオーバーレイ描画する機能を
追加する。レート履歴はフロント側でスナップショット更新ごとに ring buffer に
蓄積する（サーバ側変更なし）。

**制約**
- 依存追加なし。vendor 済みライブラリ（cytoscape/dagre）と素の Canvas のみ。
- 既存のデータフローアニメと共存すること（同じオーバーレイ canvas を共有可）。
- ノード 100 / エッジ 150 規模で 60fps を維持。波形はオフスクリーン canvas に
  キャッシュし、レート更新時のみ再描画してよい。
- expected rate を割ったトピックの波形は赤 (#f85149)、正常は既存のアクセント色。

**受け入れ基準**
- `rgd serve --demo` で、detector 停止フェーズに `/objects` エッジの波形が
  平坦化し赤変色すること。回復フェーズで元に戻ること。
- Settings タブに ON/OFF トグルがあり、OFF で描画もコストもゼロになること。
- ズームアウト時（一定スケール以下）は自動非表示になること（ゴチャつき防止）。
- `python -m pytest test/` が全部通ること。

**禁止事項**
- `index.html` の構造変更（Settings タブへのトグル 1 行追加は可）。
- スナップショット API / 録画フォーマットのスキーマ変更。
- `style.css` の既存セレクタの変更（追記は可）。

**レビュー観点（Claude 用メモ）**
- ring buffer の確保がトピック数に比例して無限に増えないか（消えたトピックの掃除）。
- 再生スクラブ時（タイムライン操作）に履歴が時系列で破綻しないか。
- 文字どおり毎フレーム全波形を描き直していないか。

### T2: シネマティック・リプレイ「Incident Theater」 ← 本命

**背景**
「ロボットの事故をリプレイ映像で見る」は SNS で最も強い絵。障害発生時に
カメラが自動でボトルネックノードへパン＆ズームし、スローモーション＋字幕で
事故を再生する。タイムラインスクラバ・録画再生・pipeline-path tracer が
すでに揃っているので、足すのは「カメラワーク＋字幕＋書き出し」のみ。
完成すれば README のヒーロー GIF をワンクリックで再生成できるようになり、
以後のデモ素材製作コストがゼロになる。

**タスク**
1. **シネマモード** — `rgd serve --demo --cinema`（および UI 上のボタン）で
   起動する自動演出モードを `web/app.js` に追加:
   - 再生開始: 全体俯瞰（fit）でデータフローを数秒見せる。
   - issue 発生検知: 該当ノードへ cytoscape の `animate()` でパン＆ズーム、
     再生速度を 0.3x に落とす。
   - 字幕バー: 画面下部に issue の要約（例:
     `detector callback p95 210 ms — /objects 4.1 Hz (expected ≥ 10)`）を
     タイプライタ風に表示。
   - pipeline path を自動点灯（既存 `/api/v1/path` 相当のクライアント処理を流用）。
   - 回復検知: ズームアウトして readiness バーが緑に戻るのを見せ、
     「Recovered ✓」字幕で締める。
2. **書き出し** — UI に「Export WebM」ボタン。`canvas.captureStream()` +
   `MediaRecorder` でシネマモード 1 周を WebM 録画してダウンロード。
   （GIF 変換はユーザー側で `ffmpeg` 想定。README にコマンド例を追記。）

**制約**
- 依存追加なし。MediaRecorder はブラウザ標準 API を使う。
- シネマモードは再生(replay)時のみ有効。ライブ接続時はボタンを無効化。
- カメラワークの各フェーズ（俯瞰秒数、ズーム倍率、スロー倍率）は app.js 冒頭の
  定数オブジェクト 1 箇所にまとめ、演出調整を diff 1 行でできるようにする。
- 字幕テキストは issue データから機械生成。ハードコード禁止
  （`--demo` 以外の録画でもそのまま動くこと）。

**受け入れ基準**
- `rgd serve --demo --cinema` をブラウザで開くと、操作なしで
  俯瞰 → ズーム＆スロー＆字幕 → 回復 → 俯瞰 の 1 サイクルが自動再生されること。
- 任意の録画（`rgd serve run.rgd.json --cinema`）でも、issue を含む録画なら
  同じ演出が機械的に成立すること。issue が無い録画では通常再生に
  フォールバックし、エラーを出さないこと。
- Export WebM で 1 サイクル分の動画ファイルが落ちてくること。
- GitHub Pages デモ（完全クライアントサイド）でもシネマモードが動くこと。
- `python -m pytest test/` が全部通ること。

**禁止事項**
- 録画フォーマット・スナップショットスキーマの変更。
- サーバ側 (`server.py`) への録画機能の追加（書き出しはブラウザ内で完結）。
- 外部 CDN からのライブラリ読み込み。

**レビュー観点**
- issue の発生/回復の検知ロジックが「issue 配列の出現/消滅」だけに依存して
  いるか（特定の issue 種別にハードコードされていないか）。
- 複数 issue が同時発生したときの挙動（最初の CRITICAL を優先、など方針が
  コードに明示されているか）。
- シネマモード中のユーザー操作（クリック・スクラブ）で演出を中断できるか。

**完了後の上司タスク**: demo.gif をシネマモード書き出しで再収録し README 更新。

### T3: PR にグラフ差分画像を自動コメントする GitHub Action

**背景**
`rgd diff --fail-on-regression` で CI ゲートはすでに可能。しかし数字の羅列では
誰も見ない。**悪化した経路を赤くハイライトした before/after のグラフ画像**を
PR に自動コメントすれば、「CI がロボットの回帰を絵で教えてくれる」が
スクショ 1 枚で伝わる。他人のリポジトリの PR 上にこのツールの絵が出る＝
最強の口コミ導線。

**タスク**
1. **画像レンダラ** — `rgd diff old.rgd.json new.rgd.json --image diff.png` を
   追加。ヘッドレスブラウザに頼らず、グラフを SVG として自前レンダリング
   （`ros_graph_debugger/report.py` の HTML レポート生成と同系統の経路で、
   ノード配置は dagre 相当の単純レイヤードレイアウトを Python 側で実装）し、
   regressions のあるノード/エッジを赤ハイライト。before/after を左右並置した
   1 枚の SVG → PNG 変換は cairosvg を **optional dependency** として使う
  （未インストールなら SVG のみ出力し、その旨を表示）。
2. **Action 化** — リポジトリ直下に `action.yml` を追加。入力:
   `baseline`（基準録画のパス）、`candidate`（PR 側の録画パス）、
   `github-token`。動作: diff 実行 → 画像生成 → PR にコメント
   （画像は workflow artifact にアップロードし、コメントには Markdown
   summary + 画像リンクを貼る）。regression があれば exit 1。
3. **ドキュメント** — README に「CI で回帰を絵で検出する」節を追加し、
   コピペで動く workflow YAML 例を載せる。

**制約**
- 本体の必須依存は増やさない（cairosvg は extras: `pip install ros_graph_debugger[image]`）。
- SVG レンダラは `--demo` 録画 2 本（正常版・劣化版）で動くフィクスチャを
  `test/` に同梱し、ユニットテストで SVG 構造（赤ハイライトの有無）を検証。
- Action は composite action（Docker 不要、`runs: using: composite`）。

**受け入れ基準**
- `rgd serve --demo` 由来の録画 2 本に対し `rgd diff A B --image out.svg` で、
  劣化ノードが赤い before/after 並置 SVG が生成されること。
- regression ゼロの 2 本では「No regressions ✓」バッジ風の画像が出ること。
- このリポジトリ自身の CI に組み込み、ダミー PR で実際にコメントが
  付くことを確認（確認は上司側でやるので、workflow ファイルまで用意すれば可）。
- `python -m pytest test/` が全部通ること。

**禁止事項**
- ヘッドレス Chrome / Puppeteer / Playwright の導入。
- 既存 `diff.py` の CLI 互換性を壊す変更（フラグ追加のみ可）。

**レビュー観点**
- SVG レイアウトがノード 50 規模でも崩壊しないか（巨大グラフでは
  regression 周辺のサブグラフだけに自動で絞る、等の逃げがあるか）。
- Action がフォーク PR（token 権限が落ちる）でコメント失敗したとき、
  CI 自体は diff の exit code で正しく落ちるか。

### T4: `rgd top` — ターミナル TUI (htop for ROS 2)

**背景**
ターミナル GIF は ROS 界隈・HN/Reddit で異常に受けがいい。さらに
「SSH しかできない実機」という実需が強い。Web UI と同じスナップショットを
食ってターミナルに描くだけなので、サーバ側は変更不要。デモ動画の
2 本目の柱になる。

**タスク**
`rgd top [URL | recording.rgd.json --demo]` サブコマンドを追加。
表示要素:
- 上段: readiness バー（ステージごとの OK/WARN/ERROR を色ブロックで）。
- 中段: ノード/トピックのテーブル（rate / expected / CPU / callback p95、
  ソートキー切り替え、劣化行は赤）。各行に Unicode ブロック文字の
  ミニスパークライン（直近 30 秒のレート）。
- 下段: issue パネル（CRITICAL から順に、evidence 1 行付き）。
- キー操作: `q` 終了、`s` ソート切替、`/` フィルタ、`p` 一時停止。

**制約**
- 依存は **textual を optional dependency** とする
  （`pip install ros_graph_debugger[tui]`）。本体の必須依存は増やさない。
- データ取得は既存の HTTP API / 録画リプレイ経路を流用。TUI 用の新 API を
  生やさない。
- `--demo` で DDS なしで動くこと（この開発環境の制約でもある）。

**受け入れ基準**
- `rgd top --demo` で、detector 停止時にテーブルの該当行とスパークラインが
  赤くなり、issue パネルに bottleneck が出ること。
- 80x24 の最小端末でもレイアウトが崩壊しないこと（要素を優先度順に間引く）。
- textual 未インストール時は「pip install ros_graph_debugger[tui]」と案内して
  正常終了すること。
- `python -m pytest test/` 全パス。TUI のウィジェット構築ロジックは
  ヘッドレスでユニットテスト（textual の pilot/test 機構を使用）。

**禁止事項**
- curses 直書き（textual に統一）。
- サーバ・スナップショットスキーマの変更。

**レビュー観点**
- ポーリング間隔とちらつき（差分更新になっているか）。
- 録画リプレイ時の時間制御が Web UI のスクラバと同じセマンティクスか。

**完了後の上司タスク**: asciinema + agg でターミナル GIF を収録し README に追加。

### T5: フリートウォール（federation の管制室ビュー）

**背景**
`rgd federate --serve` で複数ロボットのマージ表示はすでにある。これを
「管制室」らしい絵にする: ロボットごとのミニグラフ＋readiness バーを
タイル状に並べ、1 台が落ちるとタイルが赤く脈動。「10 台のロボットを
1 画面で監視」の絵は企業ユースへの訴求力が別格。

**タスク**
1. federation サーブ時の Web UI に「Fleet」タブを追加。ホストごとのタイル
   （ホスト名 / readiness バー / issue 数バッジ / ノード・トピック数 /
   ミニグラフサムネイル）をグリッド表示。
2. タイルクリックで該当ホストのフルグラフにドリルダウン（既存ビューを
   ホストでフィルタ）。
3. CRITICAL issue を持つホストのタイルは赤ボーダー＋ゆっくり脈動
   （CSS アニメーション）。
4. **デモ手段**: クロスプロセス DDS が使えないため、デモは録画リプレイの
   多重化で行う。`rgd serve --demo --fleet N` で同一デモ録画を N 台分
   （ホスト名 robot1..robotN、位相をずらして）リプレイし、フリート表示を
   DDS なしで見せられるようにする。

**制約**
- 依存追加なし。
- ミニグラフサムネイルは各ホストのグラフを小型 canvas に静的描画
  （リアルタイムアニメ不要、更新は数秒に 1 回で可）。
- `federation.py` の merge ロジック・スナップショット構造は変更しない
  （フロント側でホスト名前空間をパースして分配する）。

**受け入れ基準**
- `rgd serve --demo --fleet 6` で 6 タイルのフリートウォールが表示され、
  位相ずれにより常にどこか 1〜2 台が赤く脈動している状態になること
  （= スクショを撮れば必ず映える状態）。
- タイルクリック → フルグラフ → 戻る、の導線が機能すること。
- 通常の単一ホスト表示に一切影響がないこと（Fleet タブはホストが
  複数あるときのみ出現）。
- `python -m pytest test/` 全パス。

**禁止事項**
- `federation.py` のスキーマ変更。
- 単一ホスト時の既存 UI の挙動変更。

**レビュー観点**
- タイル数 20 でも描画負荷が破綻しないか（サムネイル更新の間引き）。
- ホスト名のサニタイズ（名前空間区切りの衝突）。
